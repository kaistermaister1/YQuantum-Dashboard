01 / THE CHALLENGE
Two Algorithms. One Quantum Computer.
One Real Insurance Problem.
Which quantum approach scales better on P&C insurance product bundling?

QAOA
The hybrid variational approach: a classical optimizer repeatedly chooses circuit angles, and the quantum device estimates the cost. Well-studied, but performance depends on depth, noise, and optimizer behavior.

DQI
The fixed-circuit interference approach used in this challenge: encode the instance, run the circuit, decode, and post-select. There is no variational parameter loop, but decoding and post-selection still matter.

Your Mission
Benchmark both on the official instance sizes n = 10, 21, 20, 50, 200. Compare solution quality, workflow cost, and scaling against a classical baseline.

How to read this deck:
First understand the business objective and constraints. Then follow the ILP-to-QUBO bridge, and only then compare QAOA and DQI. Keep three metrics separate throughout: solution quality, total workflow cost, and hardware resources.

06 / QAOA
QAOA: The Variational Approach
Classical optimizer + quantum circuit in a feedback loop

The Loop
Prepare all qubits in superposition (H gates)
Cost layer: Rz for objective + ZZPhase for penalties
Mixer layer: Rx rotations explore new solutions
Measure → compute energy → optimizer adjusts angles
Repeat many objective evaluations until the estimated cost stops improving
Why This Hardware Fits QAOA
ZZPhase is native: ZZ-style penalty couplings map naturally to the hardware gate set
All-to-all connectivity: no SWAP overhead
Rz is free: virtual gate, zero physical cost
Start with p = 1, try p = 2 if time permits

07 / DQI
DQI: The Interference Approach
Error correction in reverse — one shot, no optimization loop

The Pipeline
Dicke state: equal superposition of k-hot strings
Phase encoding: Rz(ci) on each variable qubit
Syndrome encoding: CNOT network from parity matrix B
Decoder stage: optional BP1-style decoding for feasibility steering
Hadamard + Measure: interfere and read out
Post-select: keep only syndrome = 0 results
Key Tradeoffs
No variational optimizer loop: this implementation fixes the circuit structure, then decodes and post-selects
More qubits needed: n variables + m syndrome qubits
Re-encoding risk: parity constraints can lose information if the max-XORSAT model is too coarse
Post-selection cost: ~10–30% of shots kept
Based on LDPC coding theory (same math as 5G!)

Key Insight
QAOA is variational: a classical optimizer tunes quantum circuit parameters in a feedback loop.

Think of it like black-box parameter tuning: the optimizer proposes angles, the quantum computer estimates the cost, and the loop repeats.

min_{gamma,beta} <psi(gamma,beta)| H_cost |psi(gamma,beta)>
Important distinction: QAOA minimizes an expected cost, but the candidate answers are the measured bitstrings. Low expectation value usually helps, but the final answer still comes from sampling.

The quantum computer explores many states in superposition; the classical optimizer only sees noisy estimates of their average cost and steers the angles from there.

The Depth-Quality Tradeoff
The Tradeoff
More layers (higher p) can improve the best reachable objective, but they also cost more:

-- More parameters for the optimizer to tune (harder optimization)
-- Deeper circuit = more gates = more noise on real hardware
-- Diminishing returns: going from p=1 to p=2 helps a lot, p=10 to p=11 barely matters

For the challenge's package-local blocks, p=1 or p=2 is a practical starting point on near-term hardware, not a guaranteed optimum.

Section A
Why DQI Re-encodes the Problem
Start in business language
The insurance challenge begins as a binary optimization problem: each coverage decision is a bit, and the model is written using prices, discounts, family rules, dependencies, incompatibilities, and package limits.

choose x_i in {0,1}
maximize business value
subject to binary constraints
That is a normal ILP mindset: add variables, compare sums, and enforce rules with equalities or inequalities.

What DQI wants instead
DQI does not work directly with ordinary sums like x_a + x_b ≤ 1. Its circuit is built around parity checks, so it wants statements of the form:

x_1 ⊕ x_4 ⊕ a_2 = 1
This is the language of max-XORSAT: XOR clauses over bits, sometimes with helper bits added.

Why the translation matters
The DQI circuit computes a syndrome: a record of which parity checks are satisfied or violated.

If the problem is still written only as ordinary ILP sums and inequalities, there is no parity-check structure for the circuit to evaluate.

1
Original rule
Start from a binary ILP constraint written in plain optimization language.

2
Ask what type it is
Is it already a parity-like equality, or is it an inequality / logic rule that needs extra work?

3
Introduce helper bits if needed
Slack, carry, or intermediate logic bits help preserve the meaning of the original constraint.

4
Rewrite as XOR clauses
Each finished rule becomes a parity check that asks for even or odd parity.

5
Feed DQI
The clauses become the parity-check data that the quantum circuit and decoder operate on.

Rule that converts cleanly
If you must choose exactly one of two options, parity already captures the rule.

ILP: x_auto_basic + x_auto_enhanced = 1
XOR: x_auto_basic ⊕ x_auto_enhanced = 1
Good news. For two binary variables, “sum equals 1” and “XOR equals 1” mean the same thing: exactly one bit is on.

Rule that does not convert cleanly
Many realistic constraints are not parity statements by themselves.

ILP: x_gap + x_floater ≤ 1
Wrong shortcut: x_gap ⊕ x_floater = ?
Real move: add helper bits → build gadgets → get XOR clauses
Important warning. Taking an inequality “mod 2” usually loses the information you actually care about.

Big picture: DQI is not solving a different business problem. It is solving the same business problem after that problem has been rewritten into a parity-check language that the circuit can understand.

What a parity clause is saying
A clause like

x_1 ⊕ x_4 ⊕ a_2 = 1
does not mean “add these numbers in the usual sense.” It means:

look only at the listed bits,
count whether an even or odd number of them are 1,
demand the required parity on the right-hand side.
If the clause equals 0, it asks for even parity. If the clause equals 1, it asks for odd parity.

Even parity
x_a ⊕ x_b ⊕ x_c = 0 means the number of 1s among those bits must be even.

Allowed examples: 000, 011, 101, 110
Rejected examples: 001, 010, 100, 111
Odd parity
x_a ⊕ x_b ⊕ x_c = 1 means the number of 1s among those bits must be odd.

Allowed examples: 001, 010, 100, 111
Rejected examples: 000, 011, 101, 110
Student tip: When you see XOR, read it in words as “the parity of these bits.” That mental translation is much more helpful than trying to treat it like ordinary addition.
Bridge to the next section: One XOR clause is one parity check. A full DQI instance has many such clauses, and the next step is to organize them into a compact matrix form.

Section C
From ILP to max-XORSAT, Step by Step
Goal of this section: keep the original optimization problem intact while rewriting it into a parity-check form. We are changing the representation, not the business meaning.
1
Start with the original bits
Write the business decisions as binary variables x. These are the choices you actually care about, like whether a coverage is selected.

2
Convert what converts exactly
Some equalities already behave like parity rules. Those can become XOR clauses directly with no extra machinery.

3
Add helper bits when needed
Inequalities and intermediate logic often need slack, carry, or ancilla bits so the rewritten system still means the same thing.

4
Stack the clauses
Once every rule is written as a parity check, collect them into matrix form: B y = v (mod 2).

x and y
x is the original decision vector. After adding helper bits, the enlarged vector of all bits is called y.

B
Each row of B describes one parity clause. A 1 means that variable participates in the clause; a 0 means it does not.

v
The entry v_j tells you whether row j wants even parity (0) or odd parity (1).

Weights
Feasibility is not the whole story. Weights carry information about solution quality so the encoding still reflects the objective, not just valid/invalid status.

Three clauses become three rows
Suppose one helper bit a0 has been introduced and the rewritten parity checks are:

clause 1: x0 ⊕ x1 = 1
clause 2: x0 ⊕ a0 = 0
clause 3: x2 ⊕ a0 = 1
Read each line one by one. Clause 1 asks for odd parity on x0 and x1. Clause 2 asks for even parity on x0 and a0.

Now package the same information
y = [x0, x1, x2, a0]^T

B =
[1 1 0 0]
[1 0 0 1]
[0 0 1 1]

v = [1, 0, 1]^T
Nothing new happened mathematically. We just compressed “many clause equations” into one matrix statement: B y = v (mod 2).

Exact conversion
Some families in the challenge naturally reduce to parity. With two mutually exclusive choices and a requirement to pick exactly one, XOR already says the right thing.

ILP: x_auto_basic + x_auto_enhanced = 1
XOR: x_auto_basic ⊕ x_auto_enhanced = 1
Interpretation: the row of B would place 1s under those two variables, and the matching entry of v would be 1.

Helper-bit conversion
Constraints like incompatibilities and dependencies rarely become a single XOR clause on their own.

ILP: x_gap + x_floater ≤ 1
Not enough: x_gap ⊕ x_floater = ?
Real path: helper bits + gadgets → several parity clauses
Interpretation: the helper bits are bookkeeping variables that let the parity system remember the logic of the original inequality.

What students should remember
y = all bits. Start with the original decision bits and append helper bits only when needed.
B = clause membership. Each row records which bits participate in one parity check.
v = target parity. A 0 means even parity; a 1 means odd parity.
